## Phase 2: Experiment tagging and data classification schema

### 2a: New columns on the bets table (and any related prediction/trade tables)

Add the following columns:

- `data_tier` ENUM: `'experiment'`, `'candidate'`, `'promoted'` — default `'experiment'`
- `experiment_tag` TEXT (nullable) — e.g. `"turkish-super-lig"`, `"new-corners-model-v2"`, `"brazil-serie-b-btts"`
- `opportunity_boosted` BOOLEAN default `false` — flag for bets where opportunity scoring was artificially boosted to generate learning data in low-data scenarios
- `original_opportunity_score` FLOAT (nullable) — the raw opportunity score BEFORE any boost was applied
- `boosted_opportunity_score` FLOAT (nullable) — the score AFTER boost, so the model can compare organic vs boosted performance
- `sync_eligible` BOOLEAN default `false` — only `true` for `promoted` tier bets
- `promoted_at` TIMESTAMP (nullable) — when the bet's league/strategy was promoted
- `promotion_audit_id` TEXT (nullable) — links to the promotion decision record

### 2b: New `experiment_registry` table

This is the model's master list of what it's testing and where each experiment stands:

```
experiment_registry:
  id                    TEXT PRIMARY KEY (e.g. "turkish-super-lig-match-result")
  experiment_tag        TEXT NOT NULL (matches bets.experiment_tag)
  league_code           TEXT
  market_type           TEXT (e.g. "match_result", "btts", "over_under", "corners", "cards")
  data_tier             ENUM('experiment', 'candidate', 'promoted', 'demoted', 'abandoned')
  created_at            TIMESTAMP
  tier_changed_at       TIMESTAMP
  current_sample_size   INTEGER default 0
  current_roi           FLOAT
  current_clv           FLOAT
  current_win_rate      FLOAT
  current_p_value       FLOAT
  current_edge          FLOAT
  consecutive_negative_weeks  INTEGER default 0
  notes                 TEXT
```

### 2c: New `promotion_audit_log` table

Every promotion or demotion decision must be logged and auditable:

```
promotion_audit_log:
  id                    TEXT PRIMARY KEY
  experiment_tag        TEXT
  previous_tier         TEXT
  new_tier              TEXT
  decision_reason       TEXT (detailed explanation of why)
  metrics_snapshot      JSONB (full snapshot of all metrics at decision time)
  thresholds_used       JSONB (the thresholds that were applied)
  decided_at            TIMESTAMP
  decided_by            TEXT ('auto_promotion_engine' or 'manual_override')
```

### 2d: New `experiment_learning_journal` table

The model's self-analysis log:

```
experiment_learning_journal:
  id                    TEXT PRIMARY KEY
  analysis_date         TIMESTAMP
  experiment_tag        TEXT (nullable — null means system-wide analysis)
  analysis_type         TEXT ('performance_review', 'opportunity_scoring_audit', 'promotion_readiness', 'demotion_warning', 'strategy_adjustment')
  findings              JSONB
  recommendations       JSONB
  actions_taken         JSONB
```

### 2e: Auto-tagging rules

Implement the following automatic tagging logic in the trading engine:

- When a bet is placed on a league/team/market combination where the model has FEWER than 50 historical data points → automatically set `data_tier = 'experiment'` and `experiment_tag` to `"{league}-{market_type}"` format
- When opportunity scoring has been boosted above its organic level to increase the chance of a bet being placed → set `opportunity_boosted = true`, record both the original and boosted scores
- ALL bets in dev default to `sync_eligible = false` unless their league/market combination has `data_tier = 'promoted'` in the `experiment_registry`
- Bets on promoted league/market combinations in dev automatically get `sync_eligible = true` and `data_tier = 'promoted'`

## Phase 3: Promotion engine — automated, threshold-based, auditable

### 3a: Promotion thresholds (configurable via environment variables with these defaults)

**Experiment → Candidate:**
- `PROMO_MIN_SAMPLE_SIZE` = 30 (minimum settled bets in dev)
- `PROMO_MIN_ROI` = 3.0 (minimum ROI percentage)
- `PROMO_MIN_CLV` = 1.5 (minimum average closing line value percentage)
- `PROMO_MIN_WIN_RATE` = 52.0 (minimum win rate percentage — adjust per market type)
- `PROMO_MAX_P_VALUE` = 0.10 (results must be statistically significant at 90% confidence)
- `PROMO_MIN_WEEKS_ACTIVE` = 3 (must have data across at least 3 separate weeks — prevents hot streak promotion)
- `PROMO_MIN_EDGE` = 2.0 (minimum average estimated edge percentage)

**Candidate → Promoted:**
- `PROMO_CANDIDATE_MIN_SAMPLE` = 20 (additional bets at candidate-level reduced stakes in prod)
- `PROMO_CANDIDATE_MIN_ROI` = 2.0
- `PROMO_CANDIDATE_MIN_CLV` = 1.0
- `PROMO_CANDIDATE_MAX_P_VALUE` = 0.05 (tighter — 95% confidence for full promotion)
- `PROMO_CANDIDATE_MIN_WEEKS` = 2

### 3b: Demotion thresholds

**Promoted → Candidate (automatic demotion):**
- ROI drops below 0% over rolling 30-bet window in prod
- CLV drops below 0% over rolling 30-bet window
- 3 consecutive weeks of negative ROI
- Any of these triggers automatic demotion + audit log entry

**Candidate → Experiment (automatic demotion):**
- ROI drops below -5% during candidate phase
- CLV negative over candidate sample
- Immediate demotion back to experiment for more dev-side learning

**Abandoned status:**
- If an experiment has 50+ bets and ROI is below -10% with p-value < 0.10 (i.e. it's statistically significantly BAD), auto-flag as `abandoned` with a recommendation to stop testing

### 3c: Promotion engine script

Create `scripts/promotion-engine.ts` that:

1. Runs automatically as a cron job every 24 hours (after overnight learning at 3am)
2. Iterates through all entries in `experiment_registry`
3. Recalculates all metrics (sample size, ROI, CLV, win rate, p-value, edge, weekly consistency) from the actual bet data
4. Checks each experiment against the appropriate thresholds for its current tier
5. If thresholds are met → promotes to the next tier, logs in `promotion_audit_log` with full metrics snapshot and reason
6. If demotion thresholds are triggered → demotes, logs the demotion with explanation
7. Updates `sync_eligible` on all bets accordingly (promoted = eligible, everything else = not eligible)
8. Writes an entry to `experiment_learning_journal` summarising what changed and why
9. CRITICAL: The promotion engine must NEVER skip the thresholds or allow exceptions. No bet data reaches prod sync without passing through this gate.

### 3d: Candidate-tier stake reduction

When a league/market is at `candidate` tier:
- In prod, the staking engine must apply a `CANDIDATE_STAKE_MULTIPLIER` = 0.25 (25% of normal Kelly stake)
- This is hardcoded as a safety measure — the model cannot override it
- Once fully promoted, normal Kelly staking applies
- Log that candidate-level staking was applied on every candidate-tier bet

### 3e: Manual override endpoint (emergency only)

- `POST /api/admin/promote` — body: `{ experiment_tag, target_tier, reason }` — allows manual promotion/demotion with a mandatory reason that gets logged in the audit table with `decided_by = 'manual_override'`
- This is for emergencies only. The automated engine is the primary path.
