### 3c: Naming normalisation

This is CRITICAL. Different APIs use different names for the same competition:
- "Premier League" vs "English Premier League" vs "England - Premier League"
- "UEFA Champions League" vs "Champions League" vs "CL"

Create a `competition_aliases` table that maps variant names to canonical names. When ingesting from multiple sources, always normalise to the canonical name before storing.

Similarly, team names vary across APIs:
- "Man United" vs "Manchester United" vs "Manchester Utd"
- "Inter" vs "Inter Milan" vs "Internazionale"

Extend the existing team name normalisation (if any) to handle these variants. Use fuzzy matching with a confidence threshold — log any matches below 90% confidence for manual review.

## Phase 4: Experiment pipeline integration

All new competitions flow directly into the experiment pipeline:

- Every new league/market combination starts as `data_tier = 'experiment'` automatically
- The model builds statistical profiles as data accumulates
- International competitions (World Cup, continental cups) get their own experiment tags like `"fifa-world-cup-match-result"`, `"copa-america-btts"`
- Women's competitions get tags like `"wsl-match-result"`, `"nwsl-btts"`
- Cup competitions get tags like `"fa-cup-match-result"`, `"copa-del-rey-btts"`

The model treats all competitions equally in the experiment pipeline — the promotion thresholds don't care whether it's the Premier League or the Guatemalan Liga Nacional. What matters is statistical proof of edge.

### Special consideration for international competitions:

International tournaments (World Cup, Euros, Copa America etc.) have fewer fixtures per year than domestic leagues. The model needs to account for this:
- International experiment tags will take LONGER to reach 30 bets — this is expected and fine
- The `PROMO_MIN_WEEKS_ACTIVE` threshold should still apply (prevents promoting on a single tournament's hot streak)
- Consider grouping related qualifiers with the main tournament under one experiment tag IF the same prediction model and feature set applies (e.g. "fifa-wc-qualifiers-match-result" might group all confederation qualifiers)

## Phase 6: Pinnacle coverage awareness

We're at budget on Pinnacle (OddsPapi). Not all new competitions will have Pinnacle odds. The system should:

- Track which competitions have Pinnacle odds available (query OddsPapi once to check)
- For competitions WITHOUT Pinnacle odds: the model can still collect data, build models, and track its own predicted odds vs actual outcomes. It just can't calculate CLV against Pinnacle.
- For the promotion pipeline: if a competition has no Pinnacle CLV data, the promotion engine should use an alternative edge metric — e.g. the model's predicted probability vs the best available bookmaker odds, or the model's accuracy against its own probability estimates.
- Flag these competitions in the experiment lab: "No Pinnacle coverage — using model self-validation"
- These can still be promoted if other metrics (ROI, win rate, p-value, sample size) meet thresholds. CLV is the strongest signal of edge, but it's not the ONLY signal.
